# Conditioned Dataset-v5 web workflow

Open **Dataset v5** at `/dataset-v5`. The feature is offline and repository-local after the explicit Harvest **Import into Dataset** action: passive inventory does not construct a backend, contact a provider, initialize CUDA, or mutate an input.

## Flow

1. In Harvest, explicitly import each completed `spritelab.harvest.dataset-handoff.v2` into Dataset. The callback revalidates the final handoff and every raw artifact, copies rather than moves the bytes into unique managed work, rehashes the raw tree and copy, binds transaction-local grouping and pack sidecars, runs the ordinary `DatasetIntakeService` below `work/datasets/managed`, and validates the completed managed output. The callback holds the project-to-work directory chain while it works. Its derived-frame tree is created directly at `derived_sprites` through an exclusive immovable anchor inside the still-unpublished unique work root; it is never published by directory rename. Only exact held-file no-replace publication of the `dataset.<identity>` receipt makes the import visible, so a post-commit reload fault leaves no contradictory unpublished failure record and an exact retry loads the immutable receipt. Where POSIX must use a named held-file stage, it retains exactly one `.target.staging-<identity>` hard-link alias; every reader revalidates the two names as the same exact two-link inode and inventories omit only that validated alias. Extra, malformed, foreign, or changed aliases fail closed. Failed unique work remains safely unreferenced for inspection; raw Harvest bytes are never rewritten or removed.
2. Select one or more of those opaque managed Dataset imports. Preview and build revalidate the import receipt, complete copied-source and intake-output inventories, provenance sidecar/grouping, original artifact manifest and handoff, and the Harvest Dataset-import receipt. Browser requests never accept a Harvest run ID or filesystem path as candidate input.
3. Preview deterministic conditioning. Only Dataset-intake-accepted exact 32×32 PNGs with hard alpha, strict CC0-1.0/public-domain provenance, and a non-conflicting known taxonomy category are eligible. Labels are deterministic filename/relative-path-grounded, low-confidence proposals with locally recomputed pixel descriptors; they are never presented as human truth. Unknown and category-disagreement rows are excluded, and every retained low-confidence row is a mandatory independent-audit subject. Selection uses equal source quotas and deterministic per-source category round-robin; no augmentation is used.
4. Build a durable candidate. Exact byte/pixel duplicates are removed. Conservative near duplicates and normalized source parent/family groups are kept in one train/validation/test split. The output contains Phase-7 NPZ arrays, portable manifests, `semantic_v3`, conditioning vocabulary, a source-group-disjoint benchmark, provenance, coverage, duplicate/split reports, and local dataset/training-manifest/loader checks.
5. Run each independent audit through the explicit server action. The browser sends only the audit kind and literal action authorization; it cannot upload a report. The server revalidates the candidate, invokes the trusted `run_independent_audit` implementation, and durably publishes the exact report and `spritelab.audit.conditioned-run-receipt.v1` before publishing a fresh no-replace `spritelab.audit.conditioned-run-action-record.v1` under the source job. The action record binds the literal-PASS report → receipt chain, job and operation, candidate inventory, and current distinct auditor implementation. Only that terminal record makes evidence selectable. The candidate builder never emits its own PASS evidence.
6. Explicitly authorize one exact freeze once. Publication revalidates both report → receipt → action chains and populates the final content-addressed Dataset and campaign directories directly through held parents. A durable `PREPARED` job journal binds both complete inventories; a no-replace Dataset marker commits only that component; the final no-replace campaign marker cross-binds the journal, Dataset marker, and both inventories and alone authorizes the pair. An exact retry after either marker adopts only marker-bound bytes. Unknown, partial, conflicting, or foreign entries are never overwritten or removed. The Dataset activation manifest retains schema `spritelab.dataset.freeze.conditioned.v5`, all six evidence bindings, and the 2,000–3,000 image contract. Training requires and independently revalidates the authoritative pair marker and every copied report → receipt → action byte.
7. After the bounded CPU/CUDA smoke is registered through Playground, run the visible Training infrastructure audit action. Dataset-v5 passively lists only verified smoke registrations bound to the selected publication; it exposes no paths and never accepts caller-authored reports or gate verdicts. The browser refreshes the current configuration SHA-256, derives every publication identity from server state, supplies a fresh opaque operation nonce, and invokes the fixed server-managed 18-gate audit. The result is rendered as a bounded identity/verdict projection. This action neither activates configuration nor starts training.
8. Explicitly activate the exact published freeze and campaign. Activation first publishes immutable receipt, record, and `PREPARED` journal bytes. One fixed no-replace project marker then binds those documents and the exact six-key prospective configuration bytes and is the sole authority. Windows performs an exact held-handle `spritelab.yaml` compare-and-swap before publishing the marker. POSIX leaves the canonical before-bytes intact and passively loads the marker-bound overlay; a third configuration SHA-256 fails closed. Restart deterministically recovers the same prepared or committed operation, including a Windows CAS completed just before marker publication. Activation never starts training.

The browser controller runs one request at a time, ignores duplicate clicks while an action is in flight, and resets the unchecked publication/activation authorizations whenever a different job is selected.

Publication does **not** modify `spritelab.yaml`, enable `execution.allow_training`, or start training. Activation and the independent training-infrastructure audit remain separate explicit actions. A single repository-local interprocess activation/launch lock, with a fixed protocol identity, serializes activation with Training Start and Resume through marker commit. Critical immutable files use exact held-file no-replace publication. Dataset/campaign directories are created at their final content-addressed names and remain non-authoritative until the pair marker; recovery validates and adopts exact bytes rather than renaming or deleting trees. A substituted or conflicting foreign inode is refused and left untouched.

The candidate and evidence bindings include `spritelab.dataset.conditioned-code-inventory.v3`. It hashes the full recursively resolved first-party import closure, including integration-only imports and every executed parent-package initializer, and binds the controlled worker executable, environment policy, runtime dependencies, and installed dependency inventories. Label-audit and dataset-validation trust use distinct transitive auditor inventories, so changing a helper invalidates the applicable prior report and receipt.

The pathname-oriented Dataset-intake/sidecar implementation runs only in an OS-confined child. On Windows, the parent applies one inheritable Untrusted mandatory-integrity `NO_WRITE_UP` SACL to the exact newly-created, empty, unique work root before creating or copying any descendants. Launch is verify-only over the bounded link-free tree; it never recursively changes ACLs, ownership, or modes. Only three exact private stdio files and the fixed, repository-local Medium/Low probe roots receive additional individually enumerated labels. A pinned interpreter starts Low, an audited prefix lowers its primary token to Untrusted before worker imports, and a one-process kill-on-close Job prevents descendants. This protects ordinary unlabeled, Low, and Medium objects; an object deliberately granting World write access at Untrusted integrity remains explicitly outside the guarantee. Linux uses inherited directory descriptors plus Landlock and `no_new_privs`; unsupported platforms fail closed.

Worker stderr is bounded and retained only in the unique private scratch tree. It is never copied into receipts, jobs, audit reports, API responses, or other portable artifacts; durable failures contain only a controlled error code.

## Durable states and API

Build jobs live below `runs/v3/conditioned-dataset-v5/`. A persisted `RUNNING` build job with no live worker is projected as `INTERRUPTED`; retrying creates a fresh job and never deletes the interrupted artifacts. Independent audit lanes also have durable ownership: a second process is refused while the owner is live, while an orphaned `RUNNING` operation is appended to immutable history as `INTERRUPTED` before a fresh operation is claimed. Incomplete audit evidence is never selected. Browser writes use the product shell's CSRF middleware and accept no filesystem paths or URLs. Initial HTML and JSON responses are rebuilt from closed public schemas: they retain only validated opaque identities, bounded counts, fixed statuses, and exact booleans required by the UI. Source titles, raw messages/events, leases, authorization records, path-bearing artifact fields, unknown nested state, and secret-like keys remain server-side.

- `GET /dataset-v5/api/inventory`
- `POST /dataset-v5/api/preview`
- `POST /dataset-v5/api/jobs`
- `GET /dataset-v5/api/jobs/{job_id}`
- `POST /dataset-v5/api/jobs/{job_id}/cancel`
- `POST /dataset-v5/api/jobs/{job_id}/evidence`
- `POST /dataset-v5/api/jobs/{job_id}/publish`
- `GET /dataset-v5/api/jobs/{job_id}/training-audit-options`
- `POST /dataset-v5/api/jobs/{job_id}/training-audit`
- `POST /dataset-v5/api/jobs/{job_id}/activate`

The evidence endpoint accepts exactly `{"kind": "label_audit" | "dataset_validation", "explicit_action": true}` and runs the trusted auditor on the server. Caller-authored report documents and filesystem paths are rejected.

The Training-audit options endpoint returns only registered opaque smoke and registration identities that match the selected freeze and campaign. The Training-audit POST accepts exactly the candidate, publication, activation-manifest, campaign-config, campaign, current-config, registered-smoke, and fresh operation identities plus `"explicit_action": true`. It rejects uploaded reports, gates, paths, activation authorization, and Start requests.

The production policy targets 2,500 images and permits 2,000–3,000 inclusive. Smaller bounds are injectable only when constructing a service directly for isolated tests; the built-in plugin always uses production bounds, and the Training activation loader independently enforces the same range.
